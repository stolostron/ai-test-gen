# ACM-20640 RBAC UI Implementation - E2E Test Cases

## Test Case 1: Validate ClusterPermission CRD Backend Functionality

**Description:** This test case validates that the ClusterPermission CRD is properly deployed and functional, enabling the creation and management of fine-grained RBAC policies for managed clusters through the backend API.

**Setup:** Ensure ACM 2.14.0 is deployed with fine-grained RBAC preview enabled, hub cluster access, and managed cluster(s) available for RBAC policy deployment.

| Step | Expected Result |
|------|----------------|
| **Step 1: Log into the ACM hub cluster** - Access the hub cluster using credentials: `oc login <cluster-api-url> --username=<username> --password=<password> --insecure-skip-tls-verify=true` | Successfully logged into the hub cluster with admin privileges. The command should output: `Login successful. You have access to X projects...` and the user should be able to access cluster resources. |
| **Step 2: Verify fine-grained RBAC preview is enabled** - Check MultiClusterHub configuration to confirm RBAC features are available: `oc get mch multiclusterhub -n ocm -o jsonpath='{.spec.overrides.components[?(@.name=="fine-grained-rbac-preview")].enabled}'` | The command returns `true`, confirming that fine-grained RBAC preview is enabled in the ACM configuration. This indicates the RBAC functionality is available for testing. |
| **Step 3: Verify ClusterPermission CRD availability** - Confirm the ClusterPermission custom resource definition is installed: `oc api-resources | grep clusterpermissions` | The output shows the ClusterPermission resource: `clusterpermissions rbac.open-cluster-management.io/v1alpha1 true ClusterPermission`. This confirms the CRD is properly installed and available. |
| **Step 4: Create a test ClusterPermission resource** - Create a ClusterPermission for granting devops group view access to a managed cluster: `oc apply -f - <<EOF`<br>`apiVersion: rbac.open-cluster-management.io/v1alpha1`<br>`kind: ClusterPermission`<br>`metadata:`<br>`  name: test-devops-view-access`<br>`  namespace: local-cluster`<br>`spec:`<br>`  clusterRoleBinding:`<br>`    roleRef:`<br>`      apiGroup: rbac.authorization.k8s.io`<br>`      kind: ClusterRole`<br>`      name: view`<br>`    subjects:`<br>`    - apiGroup: rbac.authorization.k8s.io`<br>`      kind: Group`<br>`      name: devops`<br>`EOF` | ClusterPermission resource is successfully created. The output shows: `clusterpermission.rbac.open-cluster-management.io/test-devops-view-access created`. The system accepts the resource and begins processing the RBAC policy. |
| **Step 5: Verify ClusterPermission status** - Check that the ClusterPermission is properly processed and applied: `oc get clusterpermission test-devops-view-access -n local-cluster -o yaml` | The ClusterPermission shows successful status with `AppliedRBACManifestWork: True` condition. The status section includes a message like: `Run the following command to check the ManifestWork status: kubectl -n local-cluster get ManifestWork test-devops-view-access-xxxxx -o yaml`. |
| **Step 6: Verify ManifestWork creation** - Confirm that a ManifestWork was created to deploy RBAC to the managed cluster: `oc get manifestwork -n local-cluster | grep test-devops-view-access` | A ManifestWork is created with a name like `test-devops-view-access-xxxxx`. This shows the cluster-permission operator successfully translated the ClusterPermission into deployable manifests. |
| **Step 7: Examine ManifestWork contents** - Review the generated ManifestWork to verify correct RBAC configuration: `oc get manifestwork $(oc get manifestwork -n local-cluster | grep test-devops-view-access | awk '{print $1}') -n local-cluster -o jsonpath='{.spec.workload.manifests[0]}'` | The ManifestWork contains a ClusterRoleBinding manifest with: `kind: ClusterRoleBinding`, `roleRef.name: view`, `subjects[0].name: devops`, `subjects[0].kind: Group`. This confirms proper RBAC policy generation. |
| **Step 8: Verify RBAC deployment on managed cluster** - Check that the ClusterRoleBinding was successfully applied to the target cluster: `oc get clusterrolebinding test-devops-view-access` | The ClusterRoleBinding exists on the managed cluster with the correct configuration, granting the `devops` group `view` permissions cluster-wide. This confirms end-to-end RBAC policy deployment. |

The system properly handled individual user identity for RBAC assignment. |

| **Step 5: Examine VM RBAC deployment** - Review the deployed ClusterRoleBinding for VM permissions: `oc get clusterrolebinding vm-admins-rbac-test` | The ClusterRoleBinding exists with `roleRef.name: kubevirt.io:admin` and `subjects[0].name: cluster-admins`. ## Test Case 4: Validate RBAC Policy Management and Lifecycle

**Description:** This test case tests the complete lifecycle management of RBAC policies through ClusterPermission resources, including creation, modification, and deletion scenarios to ensure robust policy management.

**Setup:** Ensure ACM hub cluster access with test users/groups available, and clean environment without conflicting RBAC policies for the test resources.

| Step | Expected Result |
|------|----------------|
| **Step 1: Log into the ACM hub cluster** - Access the hub cluster using credentials: `oc login <cluster-api-url> --username=<username> --password=<password> --insecure-skip-tls-verify=true` | Successfully logged into the hub cluster with admin privileges. The login provides access to create, modify, and delete ClusterPermission resources for lifecycle testing. |
| **Step 2: Create initial RBAC policy** - Create a baseline ClusterPermission for testing policy lifecycle: `oc apply -f - <<EOF`<br>`apiVersion: rbac.open-cluster-management.io/v1alpha1`<br>`kind: ClusterPermission`<br>`metadata:`<br>`  name: lifecycle-test-policy`<br>`  namespace: local-cluster`<br>`spec:`<br>`  clusterRoleBinding:`<br>`    roleRef:`<br>`      apiGroup: rbac.authorization.k8s.io`<br>`      kind: ClusterRole`<br>`      name: view`<br>`    subjects:`<br>`    - apiGroup: rbac.authorization.k8s.io`<br>`      kind: Group`<br>`      name: devops`<br>`EOF` | ClusterPermission is successfully created with baseline configuration. The output shows: `clusterpermission.rbac.open-cluster-management.io/lifecycle-test-policy created`. Initial RBAC policy is established for lifecycle testing. |
| **Step 3: Verify initial deployment** - Confirm the initial RBAC policy is properly deployed: `oc get clusterpermission lifecycle-test-policy -n local-cluster -o jsonpath='{.status.conditions[?(@.type=="AppliedRBACManifestWork")].status}'` | The command returns `True`, indicating the initial RBAC policy was successfully deployed. The baseline configuration is active and ready for modification testing. |
| **Step 4: Modify RBAC policy permissions** - Update the ClusterPermission to change role from view to edit: `oc patch clusterpermission lifecycle-test-policy -n local-cluster --type='merge' -p='{"spec":{"clusterRoleBinding":{"roleRef":{"name":"edit"}}}}'` | ClusterPermission is successfully updated. The output shows: `clusterpermission.rbac.open-cluster-management.io/lifecycle-test-policy patched`. The system accepts the role change from view to edit permissions. |
| **Step 5: Verify policy update deployment** - Confirm the updated RBAC policy is redeployed with new permissions: `oc get manifestwork -n local-cluster | grep lifecycle-test-policy` | A new or updated ManifestWork reflects the permission change. The ManifestWork may show a new generation or updated timestamp, indicating the cluster-permission operator processed the role change. |
| **Step 6: Add additional subjects to policy** - Modify the ClusterPermission to include cluster-admins group alongside devops: `oc patch clusterpermission lifecycle-test-policy -n local-cluster --type='json' -p='[{"op": "add", "path": "/spec/clusterRoleBinding/subjects/-", "value": {"apiGroup": "rbac.authorization.k8s.io", "kind": "Group", "name": "cluster-admins"}}]'` | ClusterPermission is successfully updated with additional subjects. The output shows: `clusterpermission.rbac.open-cluster-management.io/lifecycle-test-policy patched`. The policy now covers both devops and cluster-admins groups. |
| **Step 7: Verify multi-subject deployment** - Check that the updated policy includes both groups in the deployed ClusterRoleBinding: `oc get clusterrolebinding lifecycle-test-policy -o jsonpath='{.subjects[*].name}'` | The output shows both groups: `devops cluster-admins`. This confirms that policy modifications with multiple subjects are properly handled and deployed to the managed cluster. |
| **Step 8: Delete RBAC policy** - Remove the ClusterPermission to test cleanup functionality: `oc delete clusterpermission lifecycle-test-policy -n local-cluster` | ClusterPermission is successfully deleted. The output shows: `clusterpermission.rbac.open-cluster-management.io "lifecycle-test-policy" deleted`. The system initiates cleanup of associated RBAC resources. |
| **Step 9: Verify complete cleanup** - Confirm that both ClusterPermission and associated ClusterRoleBinding are removed: `oc get clusterpermission lifecycle-test-policy -n local-cluster 2>&1 || echo "ClusterPermission deleted" && oc get clusterrolebinding lifecycle-test-policy 2>&1 || echo "ClusterRoleBinding cleaned up"` | Both resources are properly cleaned up. Expected output: `ClusterPermission deleted` and `ClusterRoleBinding cleaned up`. This demonstrates proper lifecycle management with complete resource cleanup upon deletion. |

|------|----------------|

The login provides access to manage RBAC policies across multiple cluster contexts through the hub. |

| **Step 4: Verify first cluster RBAC deployment** - Confirm RBAC policy is applied to the first managed cluster: `oc get clusterpermission multi-cluster-policy-local -n local-cluster -o jsonpath='{.status.conditions[?(@.type=="AppliedRBACManifestWork")].status}'` | The command returns `True`, indicating successful RBAC deployment to the first cluster.
